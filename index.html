    Localhost to Production: Building a Custom Secure Tunnel

    Exposing local development servers to the public internet securely is a classic engineering challenge. To solve this, I engineered a custom robust localhost-to-production tunnel...

    Bridging Localhost to the World

    Exposing a local development environment to the public internet securely is a classic engineering challenge, especially when working with external webhooks (like Stripe or GitHub) or demonstrating features to clients before a full deployment. To solve this, I engineered a robust localhost-to-production tunnel.


    How It Works:
    Unlike traditional port forwarding which exposes your local network to severe security risks, this tunnel operates using a secure reverse proxy architecture.
    • The Client: A lightweight agent runs on the developer's machine, establishing an outbound WebSocket or TCP connection to the remote server.
    • The Relay Server: A cloud-hosted server acts as the public endpoint. It listens for incoming public HTTP requests and multiplexes them down the established secure connection to the local agent.
    • Local Routing: The local agent receives the payload and forwards it to the local development server (e.g., localhost:3000). It then captures the response and sends it back through the tunnel.


    Security & Performance:
    Security is baked in from the ground up. All traffic passing through the tunnel is encrypted using TLS. Additionally, I implemented rate limiting and request validation on the relay server to ensure that the local machine is never overwhelmed by malicious traffic.


    This project was a fantastic exploration into network programming, socket multiplexing, and secure proxying. Building tools that improve developer experience is incredibly rewarding!

    My First Open-Source Contribution: Fixing a Z3 Segfault in Java-SMT

    Exciting Milestone: My first open-source contribution has officially been merged into the java-smt project! I successfully fixed a critical Z3 segfault issue related to BigDecimal handling...

    Exciting Milestone

    My first open-source contribution has officially been merged into the java-smt project.


    The Problem (Issue #457):
    When calling IntegerFormulaManager.makeNumber with a BigDecimal value that has a fractional part (like 3.5), the Z3 solver would segfault. This was happening because the original implementation was trying to represent fractional values as division operations in the integer theory, which the underlying Z3 native API does not handle correctly.

    Here is the simple test case that reproduced the crash: 
    @Test
public void z3Bug() {
    requireIntegers();
    IntegerFormula f = imgr.makeNumber(BigDecimal.valueOf(3.5));
}
    The Solution:
    I modified the makeNumberImpl logic inside Z3IntegerFormulaManager.java to properly catch arithmetic exceptions during conversion and handle fractional parts using Euclidean division. This maintains consistency with the behavior of other solvers in JavaSMT without causing the native Z3 API to crash.

    Grateful for the opportunity to learn, contribute, and be part of the open-source community.

    Demystifying the API Gateway: Building a Scalable Gateway with Spring Boot and Redis

    An API Gateway is the central entry point for all client requests in a microservices architecture. In my recent project, I built a highly scalable API Gateway using Java Spring Boot and Redis to handle routing, real-time analytics, rate limiting, and robust authentication. Here's a deep dive into how it works...

    An API Gateway acts as a reverse proxy that accepts all API calls, aggregates the various services required to fulfill them, and returns the appropriate result. It’s a crucial pattern in microservices that protects your internal architecture from the outside world.

    In my latest project (view on GitHub), I engineered a lightweight, high-performance API Gateway using Java Spring Boot and Redis. This gateway goes beyond simple routing; it serves as a sophisticated middleware layer offering authentication, IP blocking, rate limiting, and real-time analytics.

    Why Redis?
    Redis is incredibly fast and operates entirely in-memory, making it the perfect candidate for storing session data, caching responses, and handling rate-limiting counters. By offloading these operations to Redis, the gateway can process thousands of requests per second with sub-millisecond latency without overwhelming a traditional SQL database.

    Core Features Implemented:
    • Dynamic Routing: Intelligently forwards requests to the correct downstream microservice based on URL patterns.
    • Authentication & Authorization: Validates JWT tokens at the gateway level. If a token is invalid, the request is blocked before it ever reaches a microservice, saving valuable compute resources.
    • Rate Limiting: Uses Redis-backed sliding window algorithms to prevent API abuse and DDoS attacks by limiting the number of requests a single IP or user can make in a given timeframe.
    • Real-time Analytics: Logs request metrics to monitor traffic patterns and detect anomalies instantly.
    Dockerization and Deployment
    To ensure the gateway is cloud-ready and easy to deploy, I containerized the entire application. You can easily pull and run the pre-built image directly from Docker Hub (shivammm21/apigateway): 
    docker pull shivammm21/apigateway:latest
docker run -d -p 8080:8080 --name my-api-gateway shivammm21/apigateway
    This project was a fantastic deep dive into reactive programming and distributed systems design. By centralizing cross-cutting concerns like security and monitoring into the API Gateway, the downstream microservices can remain lightweight, focused, and loosely coupled.

    My Java Spring Boot Journey: From Monoliths to Microservices

    When I first started building backend systems, everything lived inside a single, massive codebase. It worked, but as the applications grew, scaling and maintaining them became a nightmare. That's when I discovered Java Spring Boot and the power of cloud-native architectures...

    When I first started building backend systems, everything lived inside a single, massive codebase. It worked, but as the applications grew, scaling and maintaining them became a nightmare. That's when I discovered Java Spring Boot and the power of cloud-native architectures.

    Transitioning to microservices completely changed how I approach backend development. I began breaking down monolithic applications into smaller, independent services that could be developed, deployed, and scaled autonomously. Using Spring Cloud API Gateway, I centralized routing, security, and rate-limiting. I integrated Eureka Naming Server for dynamic service discovery, ensuring that services could seamlessly locate and communicate with each other without hardcoded IPs.

    Database Connections & JPA
    A critical part of this journey was mastering database connections and ORM using Spring Data JPA. Instead of writing boilerplate JDBC code, I leverage JPA interfaces to interact with my MySQL databases seamlessly.

    Here is a snippet showing a standard application.properties configuration for a robust MySQL connection: 
    spring.datasource.url=jdbc:mysql://localhost:3306/microservice_db?useSSL=false&serverTimezone=UTC
spring.datasource.username=root
spring.datasource.password=securepass

# Hibernate Properties
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MySQL8Dialect
    Once the connection is established, defining an entity and its repository is incredibly clean. Here is an example of how I define a User entity mapped to a database table: 
    @Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(nullable = false, unique = true)
    private String email;
    
    // Getters and Setters
}
    And the repository interface that provides full CRUD functionality out-of-the-box without a single line of SQL: 
    @Repository
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByEmail(String email);
}

By implementing robust authentication and authorization mechanisms across the API Gateway and individual services, I ensured that data remained protected at every layer. Coupling this with asynchronous messaging (like Kafka) and distributed tracing (like Zipkin) allowed me to build systems that are not just fast, but highly resilient and observable.
